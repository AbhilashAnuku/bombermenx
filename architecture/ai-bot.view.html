Architecture / AI bot
non-random · deterministic

AI bot — the non-random fallback player.

The brief requires that every client offer an AI that takes over the bomberman and does not behave randomly. Our bot is a small, deterministic decision tree on top of three classic algorithms: A* for pathing, BFS for blast-reach, and a threat-score map computed every tick. It is intentionally simple — it must run inside the per-tick budget alongside the simulation — but the layered structure means new behaviours plug in without changing what already works.

1 · Where it runs

The bot is a pure function of the current WorldSnapshot + the bot's own seat id. It lives in com.bombermenx.client.ai.BotPolicy and outputs an InputFrame identical to one a human keyboard would produce. The server doesn't know whether a player is human or AI — both go through the same input pipeline.

This is deliberate. It keeps the server simple, makes the AI testable in isolation against canned snapshots, and means switching to AI mid-match is a two-line client change instead of a server feature.

2 · The threat map

Before any decision, the bot builds a threat-score grid the same size as the arena (13×13 cells in the default variant). Each cell gets a numeric score from 0 (safe) to 4 (lethal this tick). The grid is recomputed every tick because bombs tick down and chains can shift the danger landscape between frames.

Cell score is the max of three contributions:

  • Bomb proximity. For every bomb on the field, BFS along the four directions out to its current radius. Score = max(0, 4 - fuseTicks/8 - distance), capped at 4.
  • Active explosions. Cells currently inside an Explosion.tiles set score 4 outright (lethal this tick).
  • Predicted chains. For each bomb already inside another bomb's radius, the “effective radius” expands. Add 1 to every cell newly inside the predicted union.

The grid is symmetric — same data structure on Vāyu and Agni sides — so it doesn't encode any side-specific tactics.

3 · The decision tree

Every tick, the bot walks this tree top-to-bottom and emits the first matched leaf as its InputFrame. Top conditions are the most urgent (survival) and bottom ones are the most discretionary (positional play).

A. Am I standing on a cell with threat ≥ 3?
Yes → escape mode. Run BFS to find the nearest cell with threat = 0 reachable within the bomb's remaining fuse. Step toward it on this tick.
A.1 No safe cell reachable in time?
Step toward the lowest-threat reachable cell instead. Better to take damage in one tick than die in two.
B. Is a bomb about to chain into my row/column?
Run the chain-prediction pass. If yes, treat as case A even though my current threat is still < 3.
C. Is there a pickup within A*(distance ≤ 5) that I can reach safely?
A* on the threat grid using cost = 1 + threat[cell]. Pickups score higher than empty cells in the heuristic. Step along the computed path.
D. Is there a destructible wall adjacent to me with no other bomb nearby?
Place a bomb (yields a pickup ~30% of the time on default tables), then immediately switch to escape mode for the next 8 ticks.
E. Is an enemy reachable within A*(distance ≤ 6)?
Approach the enemy along a path that stays at threat ≤ 1. If we reach a cell that traps them — one of their escape routes blocked by a wall — place a bomb.
F. Else: wander.
Pick the lowest-threat reachable cell within distance 3 that we haven't visited in the last 12 ticks. Avoids the bot getting stuck pacing one corridor.

4 · A* on the threat grid

The bot's search is plain A* with one twist: the per-edge cost is 1 + threat[targetCell] instead of a flat 1. That means the bot prefers to walk through safe cells even if the path is longer.

// Pseudocode — runs in ≤ 0.4 ms on a 13×13 grid
PriorityQueue<Node> open;     // ordered by f = g + h
Set<Cell> closed;
open.add(new Node(start, 0, heuristic(start, goal)));
while (!open.isEmpty()) {
    Node n = open.poll();
    if (n.cell == goal) return reconstruct(n);
    if (!closed.add(n.cell)) continue;
    for (Cell nb : neighbours4(n.cell)) {
        if (blocked(nb)) continue;        // wall / bomb / other player
        double g = n.g + 1 + threat[nb];  // <— danger-aware cost
        double f = g + heuristic(nb, goal);
        open.add(new Node(nb, g, f, n));
    }
}
return null; // no path

The heuristic is Manhattan distance — admissible because each move costs at least 1. The 13×13 closed-set is bounded so the queue stays small; we measured p99 = 0.4 ms on the reference machine.

5 · BFS for blast reach

Used in two places: during threat-map construction (how far does each bomb's explosion reach right now?) and during escape planning (which cells are reachable within the bomb's remaining fuse?). Same algorithm, two callers.

// BFS bounded by maxSteps so escape planning stays cheap
Queue<CellWithDistance> q;
Set<Cell> seen;
q.add(new CellWithDistance(start, 0));
while (!q.isEmpty()) {
    var n = q.poll();
    if (n.d > maxSteps) break;
    if (!seen.add(n.cell)) continue;
    visit.accept(n);                       // record distance
    for (Cell nb : neighbours4(n.cell)) {
        if (blocked(nb)) continue;
        q.add(new CellWithDistance(nb, n.d + 1));
    }
}

6 · Why not random?

The brief explicitly bans random AI. Random behaviour also hurts the developer experience — a deterministic bot is testable: given a seed and a starting snapshot, the bot produces the same input stream every time. That property is what lets us write integration tests like “in this exact scenario the bot must place a bomb on tick 47.”

The bot does have one randomised behaviour — when two cells have identical lowest threat and identical distance, it picks one using a per-bot PRNG seeded at match start. That tie-break is still deterministic given the seed, and prevents the bot from always picking the lexicographically-first cell, which becomes obvious to opponents within seconds.

7 · Extension hooks

Future improvements plug in without changing the tree:

  • Threat contributors. The threat map is a list of contributing functions; new tactics (e.g. predict enemy bomb placement) add a function, they don't change the existing ones.
  • Personalities. A BotProfile record holds weights — aggressive bots scale their pickup-cost down, defensive bots scale their threat-cost up.
  • Multi-step planning. A second pass on top of the decision tree could expand the search to N-tick lookahead. The single-step output is the baseline that always works.
