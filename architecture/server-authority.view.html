Architecture / Server authority & sync
60 Hz · p99 < 100 ms

Server authority & real-time synchronisation.

BomberMen-X is server-authoritative: the server runs the only canonical simulation, the client predicts locally for responsiveness, and reconciliation closes the gap whenever they disagree. This page is the contract for that loop — tick budget, prediction window, interpolation buffer, and the anti-desync rules that keep eight bombermen on the same grid.

1 · The 60 Hz tick loop

The server's match thread runs a fixed-step loop at 60 Hz (16.67 ms per tick). Every tick produces exactly one authoritative WorldSnapshot.

1
Drain inputs
Pull every queued MovePacket / BombPlacePacket per player. Reject out-of-order seq.
2
Resolve intents
Apply moves to GameWorld. Cell-locks, wall collisions, pickup pickups.
3
Advance entities
Tick bomb fuses, explosion lifetimes, pickup spawns. Chain reactions in topological order.
4
Detect end
≤ 1 alive or time expired? Transition ACTIVE → ENDED.
5
Broadcast
Serialise SyncPacket + any ExplosionPackets and emit to every connected client.

Why 60 Hz: it's the minimum rate at which a hand-input bomb placement feels “instant”. The simulation is deterministic in this order — any client given the same seed and the same input stream will compute the same outcome — which is the foundation that makes replay-from-seed tests possible.

2 · Client prediction

The client doesn't wait for the server to confirm a move. It applies the move locally on the same input frame it's sent, then keeps the move in a small ring buffer (the “pending input buffer”) until the matching snapshot arrives.

// Client per-input-frame
InputFrame f = sampleInput();              // dir + place-bomb
inputBuffer.push(f);                       // for reconciliation later
gameClient.send(new MovePacket(seq++, f));
localWorld.applyInput(f);                  // prediction
arenaRenderer.draw(localWorld);            // render the predicted state

The pending buffer has a fixed size of ~30 frames (~500 ms at 60 Hz). Anything older is dropped — at that point the snapshot loop should have caught up anyway, and replaying half a second of inputs in one frame would be wasted work.

3 · Reconciliation

Each arriving SyncPacket carries a server tick number. The client compares the snapshot to its own predicted state at the same tick:

  • Match. Trim the input buffer up to and including that tick. Render the snapshot as-is.
  • Mismatch. Replace local state with the authoritative snapshot, then replay every input still in the buffer (tick > snapshot.tick) on top of it. The replayed state becomes the new local state.

The replay is fast — at most ~30 frames of pure simulation, no rendering — and resolves the disagreement before the next render frame.

4 · Interpolation buffer

Between snapshots, the renderer interpolates linearly between the two most recently received snapshots so motion is smooth even when the network drops a frame. The interpolation cursor is kept ~100 ms behind the latest authoritative tick — a one-frame jitter cushion small enough not to feel laggy on a same-region server.

// Renderer per-frame (60 FPS)
double renderTime = serverTime - 100ms;
WorldSnapshot a = newerSnapshot;
WorldSnapshot b = olderSnapshot;
double alpha = (renderTime - b.t) / (a.t - b.t);
for (each entity) draw(lerp(b.entity, a.entity, alpha));

This buffer only affects what the renderer shows — not what the client predicts or what the server treats as truth. It exists purely to smooth visuals.

5 · Event broadcasting

Two channels share the WebSocket: state and events.

  • State — SyncPacket, 60 Hz, delta-coded against the last acknowledged snapshot when bandwidth budget tightens. The server tracks each connection's last-ack via a small per-connection ack channel piggy-backed on heartbeats.
  • Events — ExplosionPacket, MatchStatePacket, LobbyPacket, fired on transitions only. These are idempotent by id (e.g. each explosion has an integer id); clients dedupe.

The server never tries to recover dropped events. Each event carries enough context that re-deriving it from the next snapshot is cheap on the client — e.g. an explosion is purely cosmetic; the kill it caused already lives in the next snapshot.

6 · Anti-desync invariants

Five invariants the simulation guarantees:

  1. Determinism. Given a seed and an input log, every replay produces an identical state stream. Tests exercise this on every PR.
  2. Server is sole writer. No client can mutate GameWorld on its own. Predictions are local copies discarded on reconciliation.
  3. Monotonic time. The server's simulation clock is System.nanoTime, not wall clock. Resistant to NTP jumps.
  4. Bounded replay. The client's input buffer is capped at 30 frames; older entries are dropped. Replay cost is bounded.
  5. Idempotent events. Every event id is unique within a match; a client that processes the same id twice is a no-op.
