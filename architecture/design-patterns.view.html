Design patterns in BomberMen-X.

Five patterns earn their place in the codebase. Each one is here because removing it would either duplicate code or break the layering rule — never because “a pattern was applied.” This page lists them, the class that embodies each, and the alternative we rejected.

creational · GoF

Factory Method — spawning pickups, bombs, and explosions

When the simulation needs a new entity, it goes through a factory that knows the right constructor and the rules for IDs, lifetimes, and tile-locks.

Where
com.bombermenx.core.world.EntityFactory (private to GameWorld).
What it makes
Bomb, Explosion, PowerUpItem. Each carries a server-issued long id drawn from a single monotonic counter.
Why this, not new
ID monotonicity + tile-lock enforcement need to be in one place. Doing them at every call site duplicates rules and invites desync.
Rejected
Direct new Bomb(...) in MatchSession — would scatter the “next ID” counter across the file.
// Inside GameWorld — only place new Bomb / Explosion are ever called
long id = ids.next();
Bomb b = factory.bomb(id, x, y, owner.power(), fuse);
bombs.add(b);
tiles.lock(x, y, TileLock.BOMB);
events.emit(new BombPlaced(id));
behavioural · GoF

Observer — per-match event bus

The simulation emits typed events (BombPlaced, BombDetonated, PlayerKilled) on a local bus. Subscribers — the snapshotter, the moderator, the analytics sink — react without the simulation knowing they exist.

Where
com.bombermenx.core.event.MatchEventBus. One instance per MatchSession; never shared across matches.
Publishers
GameWorld exclusively. Events are typed records; no Object payload.
Subscribers
Snapshotter (turns events into delta-coded snapshot deltas), MatchModerator (kill-feed + chat filtering), MetricsCollector (Prometheus counters).
Why this, not direct calls
Lets the simulation stay pure — no knowledge of how snapshots, moderation, or metrics happen. Tests can subscribe a recorder and assert on the event stream.
events.subscribe(BombDetonated.class, e -> {
    snapshotter.markDirty(e.tilesAffected());
    metrics.bombsDetonated.inc();
    if (e.killedCount() >= 3) moderator.flagChainKill(e.id());
});
behavioural · GoF

State — match lifecycle and entity FSMs

The match itself, and several entities inside it, have discrete states with rules about valid transitions. Modelling them as a State machine keeps the “is this transition allowed?” logic in one place.

Match states
WAITING → STARTING → ACTIVE → ENDING → ENDED. Transitions live in MatchSession.transitionTo with a guard table.
Bomb states
ARMED → TICKING → EXPLODING → SPENT. The fuse tick drives TICKING → EXPLODING; an inbound blast can drive TICKING → EXPLODING early (chain reaction).
Player states
JOINING → ALIVE → SPECTATING → LEFT. Dead players transition to SPECTATING, never back.
Where
com.bombermenx.core.state.MatchState + BombState + PlayerState enums, each with a can(MatchState next) guard.
Rejected
Boolean fields (isActive, isAlive, isSpectating) — impossible-to-reach combinations (isAlive=false && isSpectating=false) make tests guess what's legal.
public enum BombState {
    ARMED, TICKING, EXPLODING, SPENT;

    public boolean can(BombState next) {
        return switch (this) {
            case ARMED    -> next == TICKING;
            case TICKING  -> next == EXPLODING;
            case EXPLODING-> next == SPENT;
            case SPENT    -> false;
        };
    }
}
behavioural · GoF

Strategy — input source and AI personalities

The simulation accepts an InputSource abstraction; humans, network inputs, and AI bots all implement the same interface. Swapping in is a one-line change.

Interface
InputSource with InputFrame next(WorldSnapshot, long tick).
Implementations
KeyboardInputSource (client), GamepadInputSource (client), NetworkInputSource (server — drains the per-player queue), BotPolicy (deterministic AI).
Where the swap happens
On the client, ArenaView.attachInputSource is the seam. On the server, MatchSession.bind(playerId, source) takes any implementation.
Why this, not subclassing
Keyboard, gamepad, network and AI have nothing in common in their internals; only the output shape (InputFrame) matters. Composition is the cheaper bond.
public interface InputSource {
    InputFrame next(WorldSnapshot world, long tick);
}

// Switch a player to AI mid-match — no other code change.
matchSession.bind(playerId, new BotPolicy(seed));
behavioural · GoF

Command — client-side input buffer for reconciliation

Every input the client sends is also stored as a Command in a small ring buffer. When the server's snapshot diverges from the client's prediction, the buffered commands are replayed in order on top of the authoritative state.

Command
InputFrame — carries direction, place-bomb flag, and client tick number. Immutable record.
Invoker
ReconciliationEngine — on every snapshot, trims acknowledged commands and re-applies the rest.
Receiver
The local copy of GameWorld. Apply is pure — same input, same world, same outcome.
Why this is the pattern
Reconciliation needs to undo + redo a sequence of inputs without knowing what those inputs did. That's the Command shape exactly.
void onSnapshot(WorldSnapshot snap) {
    localWorld.replaceWith(snap);                       // authoritative reset
    inputBuffer.dropUpToAndIncluding(snap.tick());      // trim acknowledged
    for (InputFrame cmd : inputBuffer)                  // replay
        localWorld.applyInput(cmd);
}
structural · bonus

Adapter — the WebSocket boundary

Netty's WebSocketFrame on the server and JavaFX's WebSocket-client API on the client both get adapted to a single WireChannel interface. Above that line, the code never sees Netty or JavaFX. Below it, the code never sees Envelope.

Server adapter
NettyWireChannel implements WireChannel; wraps a Netty ChannelHandlerContext.
Client adapter
FxWireChannel implements WireChannel; wraps a java.net.http.WebSocket.
Why
Keeps the simulation and snapshot layers free of transport-specific types. Switching to gRPC or QUIC in the future is a new adapter, not a rewrite.

SOLID by example

The patterns above land in service of the SOLID principles. The mapping isn't one-to-one but each principle has a single class that demonstrates it most clearly:

  • Single-responsibility — Snapshotter only turns simulation state into DTOs; it doesn't know about Netty or about the wire format.
  • Open/closed — BotPolicy takes a list of threat-contributing functions; new tactics extend the list without editing the policy.
  • Liskov — every InputSource implementation is interchangeable; the simulation cannot tell whether the source is a keyboard or an AI.
  • Interface-segregation — WireChannel exposes only send and onMessage; transport-specific options live on the implementation, not the interface.
  • Dependency-inversion — MatchSession depends on the InputSource abstraction, not on concrete keyboard / gamepad / network / bot classes.
