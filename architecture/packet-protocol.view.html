Architecture / Packet protocol
v1 · frozen W6

Packet protocol — BomberMen-X wire format.

The protocol carries every interaction between the JavaFX client and the Netty server. It is JSON-over-WebSocket today; the same envelope shape will be re-serialised to a binary form (Kryo or Protobuf) in a later sprint without changing the message types. All nine packet types share a single Envelope wrapper so framing, sequencing, and version negotiation stay in one place.

The brief from Prof. Becker mandates one constraint above all others: cross-group interoperability. Any practicum group's server must accept any other group's client. That is why every message on this page is a published contract, not an implementation detail.

Envelope

Every message on the wire is wrapped in a small envelope. The envelope carries the version, the message type, a monotonic sequence number from the sender, and the payload as a typed object. Client-side sequence numbers let the server reject out-of-order moves; server-side sequence numbers let the client deduplicate snapshot retransmits.

{
  "v":   1,                 // protocol version (uint8). bumped on breaking change.
  "t":   "MOVE",            // MessageType enum tag (UPPER_SNAKE).
  "seq": 1287,              // monotonic sequence number from the sender (uint32).
  "ts":  1716392452100,     // sender's epoch-millis (uint64). server uses for RTT only.
  "p":   { ...payload... }  // typed payload — schema differs per type.
}

Validation rules apply to every envelope: v must equal the negotiated version (mismatch → DISCONNECT); t must be a known MessageType; seq must be strictly greater than the last accepted seq from the same connection; ts must be within ±30 s of server clock or the message is dropped silently.

Message catalogue (9 types)

PlayerJoinPacket C → S

First frame after the WebSocket upgrade. Carries the player's display name, age class, and protocol version. The server replies with a LobbyPacket describing the current lobby state, or a DisconnectPacket if validation fails.

{ "t":"JOIN", "p":{
    "name":  "vayu_putra",
    "age":   "ADULT" | "CHILD",
    "vmin":  1,           // lowest acceptable protocol version
    "vmax":  1            // highest acceptable protocol version
}}
name 3–16 chars opt token 1 / connection
MovePacket C → S

Direction the player wants to move on the next tick. Re-sent each input frame even when held — keeps the wire stateless and idempotent so a dropped packet costs at most one tick of latency.

{ "t":"MOVE", "p":{
    "dir":  "N" | "S" | "E" | "W" | "NONE",
    "tick": 4271          // client's local tick guess; used for reconciliation
}}
dir ∈ enum ≤ 60 Hz strict seq
BombPlacePacket C → S

Request to drop a bomb on the current cell. Server validates that the player has at least one bomb in their pool and that the cell is unoccupied. Rejected silently — the next SnapshotPacket tells the truth.

{ "t":"BOMB_PLACE", "p":{
    "tick": 4271,
    "x":     6,            // server still re-derives from player.x
    "y":     8
}}
pool ≥ 1 cell empty ≤ 4 / s
ExplosionPacket S → C

Emitted on the tick a bomb detonates. Carries every tile in the cross-shaped blast plus IDs of any entities eliminated. Clients use this to drive particles, camera shake, and sound — pure cosmetic; the authoritative state already arrived in the next snapshot.

{ "t":"EXPLOSION", "p":{
    "id":     8101,
    "tiles":  [[6,8],[5,8],[7,8],[6,7],[6,9]],
    "killed": ["p_42","p_17"],
    "tick":   4275
}}
on event idempotent by id
MatchStatePacket S → C

Lifecycle frame: WAITING → STARTING → ACTIVE → ENDED. Clients drive scene transitions off this; nothing else opens or closes a match. Carries the active arena and the player roster.

{ "t":"MATCH_STATE", "p":{
    "state":  "ACTIVE",
    "match":  "m_2025_05_22_07",
    "arena":  "vayu_temple",
    "roster": ["p_42","p_17","p_03"],
    "endsAt": 1716393052100
}}
on transition
LobbyPacket S → C

Server's view of the lobby — list of seated players, the chosen arena variant, the countdown until match start. Re-sent on any seat change.

{ "t":"LOBBY", "p":{
    "seats": [
      { "id":"p_42", "name":"vayu_putra", "ready": true },
      { "id":"p_17", "name":"agni_sena",  "ready": false }
    ],
    "variant": "A+B",
    "secondsToStart": 14
}}
on change
SyncPacket S → C

The authoritative snapshot. ~60 Hz, delta-coded against the last acknowledged snapshot when bandwidth budget tightens. The client's reconciliation step replays buffered inputs on top of this frame whenever its prediction diverges from the truth.

{ "t":"SYNC", "p":{
    "tick":      4271,
    "players":   [{ "id":"p_42", "x":6, "y":8, "hp":1, "bombs":2, "power":3 }],
    "bombs":     [{ "id":7100, "x":6, "y":8, "fuse":52 }],
    "pickups":   [{ "id":9900, "x":2, "y":4, "kind":"RADIUS" }],
    "tiles":     "...rle-encoded delta..."
}}
≈ 60 Hz delta-coded
HeartbeatPacket C ↔ S

Liveness ping. Each side sends one every 5 s; failure to receive any frame for 30 s triggers the brief's “detonate the laggard” rule on the server, and a reconnect-with-resume attempt on the client.

{ "t":"HEARTBEAT", "p":{
    "rttHint": 42        // last observed one-way RTT in ms (optional)
}}
0.2 Hz
DisconnectPacket C ↔ S

Clean teardown frame. Carries a reason code so log lines and reconnect logic stay distinct from accidental TCP resets. Acknowledging this frame is optional; the WebSocket close immediately follows.

{ "t":"DISCONNECT", "p":{
    "reason": "VERSION_MISMATCH" | "KICKED" | "QUIT" | "DETONATED" | "AFK"
}}
reason ∈ enum

Serialization

v1 uses JSON over text WebSocket frames. This was a deliberate W3 decision so that the protocol is debuggable from a browser dev-tools panel and so that cross-group interop with other practicum groups doesn't fight over binary layouts on day one.

v2 will introduce Kryo as an optional binary mode, negotiated in the JOIN handshake via vmin / vmax. JSON stays as the lowest common denominator. The envelope shape does not change between v1 and v2 — only the bytes-on-wire encoding does.

Latency & synchronisation strategy

Three techniques apply, in this order:

  • Server authority. The server runs the only canonical simulation. Clients send inputs, the server resolves them, and the server's snapshot is the only source of truth.
  • Client prediction. The client applies its own input locally as if it were already acknowledged. The buffered inputs (last ~30 frames) are kept until the corresponding snapshot arrives.
  • Reconciliation on diff. When the next snapshot arrives, the client compares the predicted state against the authoritative one. If they match, the input buffer is trimmed. If they don't, the local state is reset to the snapshot and the buffered inputs are replayed on top, frame by frame.

Between snapshots, the renderer interpolates linearly between the two most recently received snapshots so motion is smooth even when the network drops a frame. The interpolation buffer is intentionally ~100 ms behind the authoritative tick — a one-frame jitter cushion that's small enough not to feel laggy on a same-region server.
