BomberMen-X

A server-authoritative arena bomber in Tron neon.
Architects: AA (Delivery Lead)  ·  SK (UI/UX Director)  ·  JC (Gameplay Director)

The problem we set out to solve

  • Classic arena bombers feel great offline and miserable online — input lag, desync, rubber-banding.
  • Most modern reimaginings need a hefty install or an account.
  • Accessibility is treated as a checkbox: motion, contrast, and rumble are afterthoughts.
  • We wanted a same-grid, low-latency match you can join in under sixty seconds.

Vision and pillars

Server-authoritative

Every tile, every bomb, every kill decided server-side at 60 Hz.

Readable always

Tron palette, AA contrast, motion you can dial down.

Zero install path

Browser play.html as a first-touch surface.

  • Future contributors — read our code in a week, not a month.

Three-module architecture

+-----------------+ WebSocket / JSON +-----------------+ | bomb-client | <------------------------------> | bomb-server | | (JavaFX 21) | 60 Hz tick stream | (Java 21, Netty)| +-----------------+ +-----------------+ ^ | | REST / static v | +-----------------+ +------------------------------------------- | bomb-shared | | (wire + models)| +-----------------+

Shared module = single source of truth for messages. No drift between client and server.

The wire protocol

  • HELLO / WELCOME — handshake, protocol version, session id.
  • LOBBY_STATE — roster, mode votes, countdown seconds.
  • TICK — authoritative world snapshot at 60 Hz.
  • INPUT — client intents, not positions.
  • HAPTIC — rumble cues, server-driven only.
  • CHAT, KILL, END — UI events with explicit lifetimes.

Server-authoritative tick

Deterministic

Fixed 16.67 ms step. Inputs ordered by arrival, ties broken by player id. The same inputs produce the same world.

Why it matters

Replay, anti-cheat, and bot training all become tractable. Clients only render; they never decide.

Matchmaker

  • Warmup — 20 s lobby window with mode vote.
  • Bot fill — if fewer than 4 humans, fill with bots so the match still starts.
  • Mode vote — majority wins; ties default to FFA.
  • Reconnection — 30 s grace window keyed by session id.

Game modes

Free-for-all

Last bomber standing. 4 to 8 players. Sudden-death wall closes in after 90 s.

Teams

Cyan vs Magenta. Respawn after 5 s. Round-based.

King of the Grid

Hold the centre tiles. Score ticks per second of control.

Levels

Hand-authored boards with hazards and one-way tiles.

UI / UX design language

  • Tron palette, roles fixed in tron.css: cyan primary, magenta opposing, amber warn, green success.
  • Three type roles — Display / Body / Mono — with explicit JavaFX classes.
  • 4-based spacing scale, no freelance values.
  • Motion is information: shake means a bomb, glow means urgency.
  • WCAG AA contrast on every text pair we ship.

The Tron renderer

  • ArenaRenderer — single AnimationTimer at 60 FPS on a canvas.
  • Draw order: tiles → bombs → explosions → players → particles.
  • ParticleSystem — pre-allocated ring, 600 ms TTL.
  • CameraShake — 6 px amplitude, 200 ms decay on explosions.
  • PostFx — scanlines + chromatic aberration, opacity user-controllable.

Audio + haptics

Audio

AudioBus master + SpatialAudio per source. Low thump for bombs, high ping for pickups. Default 60% master.

Haptics

HapticsService consumes server HAPTIC messages. Short pulse on hit, long on death.

Safety

  • Age gate — AgeGate.prompt(...) on first launch, answer persisted to Preferences.
  • Profanity filter — applied to display names and chat, server-side.
  • Reduce-motion — scanline opacity off / subtle / full.
  • Default audio — conservative 60% to protect headphone users.

CI/CD pipeline

  • GitHub Actions on every push: build, unit tests, integration tests.
  • Lint and contrast checks on tron.css and the presentation/ HTML.
  • Tagged release builds publish a runnable JAR and a Docker image.
  • Deploy step gated by a manual approval — the three architects sign.

Security review summary

  • Server validates every input against the current world; no trust in client positions.
  • Rate limiting on INPUT and CHAT per session.
  • Display names sanitised at the boundary; chat passes the profanity filter.
  • TLS on the public endpoint, plaintext only on explicit LAN ws://.
  • No PII collected; the age gate stores a boolean.

What worked, what we'd change

Worked

  • Three-module split kept ownership clean.
  • Shared wire module killed drift.
  • Style guide-first approach prevented a CSS pile-up.

Would change

  • Start the security review in week one, not week two.
  • Write the integration harness before the third game mode.
  • Pick the gamepad library on day one — JInput surprised us.
