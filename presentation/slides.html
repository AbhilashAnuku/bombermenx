Problem

A distributed multiplayer arena, built as a capstone

  • Build a Bomberman-style real-time game that exercises every architectural concern: domain modelling, authoritative simulation, lossy state replication, real-time UI, deployment.
  • Conform to the course supervisor's entity specification agreed at the Week 3 review (12 entities).
  • Stand up locally with one command and on a containerised target.
  • Document the decisions in arc42 shape, defend them in a 15-minute viva on Tuesday 02 June 2026.
  • Constraint: eight calendar weeks · three architects · no prior real-time multiplayer experience.

Approach

Three-tier client / server with a shared core

Shared core

bomberman-core holds every domain entity, the wire DTOs, and the codec. Compile-time guarantees that server and client cannot drift apart.

Authoritative server

bomberman-server owns the only mutable GameWorld, ticks at 60 Hz, emits WorldSnapshot frames. No client prediction.

JavaFX client

bomberman-client renders snapshots on a single Canvas; HUD reads the same snapshots. Mandala palette throughout.

Conservative scope

No persistence, no client prediction, no 3D spectator in v0.2. Each deferral was agreed with the supervisor at the Week 3 review.

Runtime

60 Hz authoritative tick, snapshot stream

  • Tick. MatchSession drains INPUT_FRAME queue, applies inputs to GameWorld, advances bomb fuses and explosions, processes pickups.
  • Snapshot. Snapshotter derives an immutable WorldSnapshot from the world; WireCodec serialises once and the server broadcasts to all match clients.
  • Bots. BotController is stepped just before input drain — structurally indistinguishable from a human at the server boundary.
  • Events. KILL_FEED and GAME_EVENT are emitted as they happen; the kill feed appears in HudOverlay.

Wire protocol

One envelope, one enum, one codec

  • Transport: WebSocket over TCP, JSON payloads, Jackson serialisation.
  • Discriminator: MessageType — 24 values: auth (4), lobby (8), match (2), gameplay (5), social (3), snapshot family (4).
  • Choke point: com.bombermenx.core.net.WireCodec — single producer and consumer of JSON.
  • Test: WireCodecTest round-trips every payload type; any DTO change that breaks parity fails the build.
  • Why JSON: text-frame readability accelerates weekly demos; per-frame compression keeps bandwidth acceptable; a future binary swap is a single-module change.

Decisions (three ADRs)

ADR-1: JSON over WebSocket

Text-frame readability accelerated debugging. WireCodec as single producer keeps a future binary swap to one module.

ADR-2: Server-authoritative simulation, no client prediction

The server holds the only mutable GameWorld. Clients render snapshots. Bots reuse the same input path as humans.

ADR-3: JavaFX desktop client (no browser target)

Team has Java expertise. Canvas renderer; SceneRouter for view changes; shaded fat-jar for distribution.

  • Client tests. TestFX coverage for menu, lobby, and HUD.
