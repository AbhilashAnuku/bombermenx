1. Executive summaryZusammenfassung

BomberMen-X is a four-player real-time arena game with a server-authoritative simulation. The system is realised as a Maven three-module Java 17 reactor — bomberman-core (shared library), bomberman-server (Netty WebSocket server, port 8080), and bomberman-client (JavaFX desktop). The architecture pursues five explicit qualities: determinism, fairness, clarity of the wire, portability, and demonstrability. Every architectural commitment in this report can be traced to one of those five goals.

BomberMen-X ist ein Vier-Spieler-Echtzeit-Arenaspiel mit serverseitig autoritativer Simulation. Das System ist als Maven-Dreimodulreaktor in Java 17 realisiert — bomberman-core (gemeinsame Bibliothek), bomberman-server (Netty-WebSocket-Server auf Port 8080) und bomberman-client (JavaFX-Desktop). Die Architektur verfolgt fünf explizite Qualitäten: Determinismus, Fairness, Klarheit der Wire-Schnittstelle, Portabilität und Demonstrierbarkeit.

The server holds the only mutable GameWorld and advances it on a fixed 16.67 ms tick budget. Clients render snapshots; they never simulate ahead of the server. Every message on the wire is a typed Envelope with a MessageType discriminator and a Jackson-serialised payload, and the codec lives in exactly one place — com.bombermenx.core.net.WireCodec. This report is the canonical document for grading and stands on its own; the long-form arc42 specification under code-explanation/architecture-spec-arc42 expands each section with class-level detail and additional diagrams.

Der Server hält die einzige veränderbare GameWorld und schreibt sie in einem festen Budget von 16,67 ms pro Tick fort. Clients zeichnen Snapshots; sie simulieren nie über den Server hinaus. Jede Nachricht auf der Wire-Schnittstelle ist ein typisiertes Envelope-Objekt mit einem MessageType-Diskriminator und einer Jackson-serialisierten Nutzlast; der Codec existiert an genau einer Stelle — com.bombermenx.core.net.WireCodec.

4. Context and scopeKontext und Umfang

The system exposes three external interfaces: the human player (keyboard or gamepad input into the client, snapshots out to pixels), the spectator role (read-only snapshot stream, deferred to v0.3), and the operator (environment variables in, Prometheus-style metrics out on port 9091). Internal dependencies are strict: bomberman-core is leaf at compile time, both bomberman-server and bomberman-client depend on it, and nothing depends back on either of them.

Das System exponiert drei externe Schnittstellen: den menschlichen Spieler (Eingabe per Tastatur oder Gamepad in den Client, Pixel hinaus), die Zuschauerrolle (read-only Snapshot-Stream, auf v0.3 verschoben) und den Betreiber (Umgebungsvariablen hinein, Prometheus-Metriken über Port 9091 hinaus). Interne Abhängigkeiten sind strikt: bomberman-core ist zur Compile-Zeit ein Blatt; sowohl bomberman-server als auch bomberman-client hängen davon ab, und nichts hängt zurück von einem der beiden.

5. Solution strategyLösungsstrategie

The architecture is the smallest configuration that satisfies the five quality goals.

Die Architektur ist die kleinste Konfiguration, welche die fünf Qualitätsziele erfüllt.

  • Three-tier client · shared core · server. A shared library (bomberman-core) defines the domain entities and the wire DTOs. The server (bomberman-server) consumes the core and owns the mutable simulation. The client (bomberman-client) consumes the same core and renders snapshots. Because both endpoints import the same record definitions, data-transfer-object drift is structurally impossible.Eine gemeinsame Bibliothek definiert Domänen-Entitäten und Wire-DTOs. Server und Client konsumieren beide denselben Kern, wodurch DTO-Drift strukturell unmöglich wird.
  • Server-authoritative simulation at sixty ticks per second.Serverseitig autoritative Simulation mit sechzig Ticks pro Sekunde. The server holds the only mutable GameWorld. Clients send InputFrame upstream and receive WorldSnapshot downstream. The server validates every input before mutation. This single decision delivers fairness, determinism, and a clean anti-cheat boundary.Der Server hält die einzige veränderbare GameWorld. Diese eine Entscheidung liefert Fairness, Determinismus und eine saubere Anti-Cheat-Grenze.
  • JSON over WebSocket. Every message is an Envelope with a MessageType discriminator and a Jackson-serialised payload. JSON is human-readable, debuggable in the browser, and small enough at sixty ticks per second for four players on the local network.Jede Nachricht ist ein Envelope mit MessageType-Diskriminator und Jackson-serialisierter Nutzlast. JSON ist lesbar, im Browser debugbar und bei vier Spielern auf einem LAN ausreichend klein.
  • JavaFX desktop client.JavaFX-Desktop-Client. A single Canvas painted by ArenaRenderer. JavaFX over a browser keeps the client a single executable artefact and shares the DTO layer with the server inside one JVM.Ein einzelnes Canvas, bemalt von ArenaRenderer. JavaFX statt Browser hält den Client als ein einziges ausführbares Artefakt und teilt die DTO-Schicht mit dem Server innerhalb einer JVM.
  • One composition root per process.Eine Komposition-Wurzel pro Prozess. Wiring happens in GameServer on the server side and ClientLauncher on the client side — plain Java construction, no dependency-injection framework.Verdrahtung erfolgt in GameServer und ClientLauncher — einfache Java-Konstruktion, kein DI-Framework.

6.2 Level 2 — ModulesStufe 2 — Module

bomberman-core contains six packages: entity/ (mutable game entities), world/ (arena and the seven concrete Bonus subclasses), sim/ (GameWorld, Snapshotter), net/ (Envelope, MessageType, WireCodec, around thirty record DTOs under net/dto/), input/ (PlayerInput), geom/ (TilePos, Direction).

bomberman-server contains nine packages: net/, session/, auth/, lobby/, match/, ai/, chat/, moderation/, config/. The composition root is GameServer; everything is wired by plain Java construction.

bomberman-client contains six packages: ui/ (the four named views and the mandala backdrop), render/, net/, audio/, input/, safety/. ArenaView repaints every 16 ms from the latest snapshot; the client holds no authoritative state.

10. Architecture decisionsArchitekturentscheidungen

ADR-001 — JSON over WebSocket as the sole wire formatJSON über WebSocket als einziges Wire-Format

Considered: JSON, binary Protobuf, binary Kryo, FlatBuffers. JSON chosen for human readability during the seven-week build budget, browser-debuggability of the protocol, and small enough payload at four players × sixty Hz on a LAN. Decision is revisited if telemetry indicates bandwidth pressure.

Erwogen: JSON, binäres Protobuf, binäres Kryo, FlatBuffers. JSON gewählt wegen Lesbarkeit im Sieben-Wochen-Budget, Browser-Debuggbarkeit und ausreichend kleiner Nutzlast bei vier Spielern × sechzig Hz im LAN.

ADR-002 — Server-authoritative simulation at sixty ticks per secondServerseitig autoritative Simulation mit sechzig Ticks pro Sekunde

Considered: client-side prediction with server reconciliation; lockstep peer-to-peer; pure server authority. Pure server authority chosen for the strongest fairness guarantee and the simplest mental model. Prediction is recorded as a v0.3 enhancement for high-latency networks.

Erwogen: Clientseitige Vorhersage mit Server-Abgleich, Lockstep-Peer-to-Peer, reine Server-Autorität. Reine Server-Autorität gewählt für die stärkste Fairness-Garantie und das einfachste mentale Modell.

ADR-003 — JavaFX desktop client rather than a browser clientJavaFX-Desktop-Client statt eines Browser-Clients

Considered: JavaFX, Swing, browser + Canvas, libGDX. JavaFX chosen for first-class OS gamepad and audio integration, a single executable artefact, and the ability to share the DTO layer in-JVM with the server.

Erwogen: JavaFX, Swing, Browser + Canvas, libGDX. JavaFX gewählt für erstklassige Gamepad- und Audio-Integration auf Betriebssystemebene.
