Abstract and contents

This document describes the architecture of BomberMan-X, a real-time multiplayer game built for the Software Architecture and Development module at SRH University Stuttgart. The descriptive sections cover the system as built; the prescriptive section records the changes we would recommend to a next team.

1. Abstract

BomberMan-X is a client and server application written in Java. The brief, by Prof. Steffen Becker, asks for a Bomberman that lets two to four players compete in one shared arena over the network. The brief also requires that any group's server must work with any other group's client. This interoperability rule is the smandalagest single constraint in the design.

The system is organised as three Maven modules. The shared library holds the deterministic simulation and the wire protocol. The server module runs the canonical match instance on top of Netty and WebSockets. The client module is a JavaFX desktop application that predicts inputs locally for responsiveness and reconciles against the server on every snapshot. The two runtime modules never depend on each other; they meet only at runtime, over the wire.

4. Descriptive: modules and layering

BomberMan-X is built as three Maven modules. The dependency edges between them are the load-bearing rule of the whole architecture. Every later diagram in this report is a different view of the same three-module structure.

4.1 The three modules

ModuleRoleMajor types
bomberman-coreShared library. Deterministic simulation, entity model, wire protocol. Compiled against by both runtimes. Depends on nothing inside the project.GameWorld, GameConfig, Bomb, Explosion, PowerUpItem, Envelope, MessageType, WireCodec
bomberman-serverAuthoritative server. Owns the only canonical GameWorld for each active match. Accepts WebSocket connections via Netty.BombServerApplication, WebSocketServer, SessionRegistry, MatchManager, MatchSession
bomberman-clientJavaFX desktop application. Predicts locally for responsiveness, reconciles against every snapshot, renders the arena and HUD.ClientLauncher, SceneRouter, ArenaView, ArenaRenderer, GameClient, BotPolicy

4.2 The layering rule

Both runtime modules depend on bomberman-core. Neither runtime module depends on the other. The two runtimes meet only at runtime, over the WebSocket wire, by exchanging the JSON envelopes defined in com.bombermenx.core.net. This rule is enforced by the Maven dependency graph; a pull request that adds an edge between the two runtimes is rejected at review.

4.3 Inner and outer layers inside the shared library

Inside bomberman-core, the inner game model (packages geom, world, entity) has no knowledge of the outer wire layer (net, net.dto). The Snapshotter class is the bridge that turns simulation state into wire DTOs. This means the simulation can run, be tested, and be replayed without opening a socket.

4.4 What this architecture optimises for

  1. Determinism. Given a seed and an input log, every replay must produce the same state stream.
  2. Cross-group interoperability. Any group's server must accept any group's client over the same envelope.
  3. Responsiveness. Client prediction keeps input feel under one frame even when round-trip time climbs.

6. Defence of the architectural choices

Five decisions shape the system. The table below names each alternative considered, the reason the chosen option won, and what would change if the other one had been picked instead.

DecisionChosenAlternatives consideredWhy this wins
Desktop client framework JavaFX 21 Swing, web client (HTML5 + Canvas) Ships with the JDK. Scene-graph plus Canvas for custom rendering. JInput-compatible. Swing's animation primitives are weaker; a web client doubles the wire-format surface area.
Server I/O Netty + WebSocket Spring Boot Web, Vert.x, plain TCP Battle-tested NIO with explicit control over the pipeline. Spring's footprint and Vert.x's reactive model fight the 60 Hz tick loop. Plain TCP would force us to re-implement WebSocket framing.
Wire format JSON over WebSocket Kryo, Protocol Buffers, FlatBuffers Inspectable in a browser dev-tools panel; no schema compiler. The cost is roughly four times the bandwidth of a binary format; the next protocol version introduces Kryo as an opt-in.
Authority model Server-authoritative Deterministic lockstep, peer-to-peer Single source of truth; clients cannot cheat; a centralised log enables replay. Lockstep is bug-prone across heterogeneous clients; peer-to-peer fragments truth and breaks moderation.
Module layout Three Maven modules One module, four or more modules Three columns map cleanly to the three runtime concerns (simulation, server, client). One module hides the layering rule; four or more adds Maven friction without a real boundary.

6.1 Tick rate, sequence numbers, and other micro-choices

QuestionChoiceReason
Tick rate60 Hz30 Hz felt sluggish on the bomb-place press; 120 Hz doubled the CPU cost with no visible benefit.
Sequence numbersMonotonic 32-bit per connectionVector clocks were over-engineered for one server per match; a single integer is sufficient.
Input buffer depth30 framesHalf a second at 60 Hz — long enough for reconciliation under typical RTT, short enough for bounded replay.
Interpolation buffer100 msOne-frame jitter cushion; smaller buffers stutter on a dropped packet, larger ones feel laggy.
Threat-map size13 × 13 cellsMatches the default arena; A* closed set fits in 169 entries, search runs in well under a millisecond.

6.2 What this report does not promise

The architecture does not target hundreds of concurrent matches per server; the brief asks for a single match. There is no horizontal scaling, no sharding, no leaderless replication. Persistence is a single Postgres instance behind the server. The cost of removing these ceilings is documented in the prescriptive section.

6. Class diagram — bomberman-core

6.1 What lives in core

bomberman-core is the shared library. It carries the deterministic game model, the value types used by both runtimes, and the on-the-wire envelope that both runtimes serialise to and deserialise from.

The central class is GameWorld. It composes the arena and every entity inside it: the active Bomb instances, the live Explosion objects, the collectable PowerUpItem entries on the floor, and the per-player PlayerState. The filled-diamond composition mark on its outgoing edges is deliberate — removing the world removes its contents.

6.2 The tick interface

The world advances through GameWorld.tick(input). The method is pure with respect to the input: given the same world hash and the same input stream, it produces the same next world. That property is what makes determinism testable.

6.3 The wire boundary

The inner game model has no reference to the outer wire layer. Three classes bridge the two sides. Snapshotter turns a GameWorld into a WorldSnapshot DTO. WireCodec serialises any Envelope to JSON. MessageType is the enum tag that says what shape sits inside the envelope.

6.4 Geometry and identity

Direction, TileType, and the id sequence generator all live in core because both runtimes need to agree on them. Player identifiers and entity identifiers are monotonic 64-bit values; the server is the only writer.

6.5 Why one library, not two

The choice to keep the simulation, the entities, and the wire types in one library — rather than split into bomb-sim and bomb-protocol — is deliberate. The simulation produces snapshots; the snapshots are the protocol payload. Splitting them would force one half to depend on the other anyway, with no real isolation benefit.

7. Class diagram — bomberman-server

7.1 The composition root

BombServerApplication is where every server-side object is wired together. It builds the Netty WebSocketServer, the in-memory SessionRegistry that tracks live connections, the MatchManager that owns active matches, and the authentication and lobby services.

7.2 From bytes to envelopes

Inbound traffic flows through Netty's pipeline up to GameServerHandler, which is responsible for parsing each WebSocketFrame into an Envelope and routing it. Move and bomb-place envelopes go to the MatchSession of the player's current match; lobby and auth envelopes go to the lobby or auth service.

7.3 Matches as state machines

Each active match is one MatchSession. The session owns the only canonical GameWorld for that match and runs the 60 Hz tick loop. The match lifecycle is held in a single MatchState enum: WAITING → STARTING → ACTIVE → ENDING → ENDED, with guarded transitions in MatchSession.transitionTo.

7.4 Authentication as an interface

AuthProvider is an interface with two implementations: DevAuthProvider for local development (any non-empty username is accepted), and GoogleAuthProvider for the production identity flow. The interface lets us swap one for the other per environment without touching GameServerHandler.

7.5 Server authority in one place

The simulation, the player roster, the bomb pool, and the pickup tables are all read and written by exactly one thread per match. Clients send intent; the server resolves it; the server broadcasts the result. There is no per-player worker thread, no per-entity actor; the match loop is the only writer.

8. Class diagram — bomberman-client

8.1 Boot sequence

ClientLauncher is the JavaFX Application entry point. It builds the SceneRouter, runs the AgeGate (child-safety check on first launch), and navigates to the MainMenuView.

8.2 Scenes and routing

A single SceneRouter owns the stage. It swaps between four views: the main menu, the lobby, the rankings page, and the in-match arena. Each view implements SceneRouter.View with two callbacks — onShown and onHidden — so that the router can start and stop per-view resources cleanly.

8.3 The render loop

ArenaView drives a JavaFX AnimationTimer at 60 FPS. Each frame, it predicts using the buffered input, calls ArenaRenderer to draw the world onto a Canvas, and lets the HudOverlay redraw on top. The renderer composes several passes: arena floor, grid lines, tiles, pickups, bombs, explosions, players, particles, and the post-effect pass (scanline + vignette).

8.4 Input and feedback

Input comes from two sources. The keyboard pump lives in ArenaView; the gamepad pump lives in GamepadPoller. Both produce the same InputFrame record. Haptic feedback (rumble) and spatial audio are emitted by HapticsService and AudioBus, based on the diff between two consecutive snapshots.

8.5 One WebSocket, many scenes

GameClient is the shared WebSocket adapter. It is created once at lobby time and stays alive across the lobby and arena scenes. Each scene subscribes to the events it cares about and unsubscribes on onHidden.

Package diagram8 / 18

9. Package diagram

9.1 The three columns

Each column is one Maven module. The columns are arranged so that the two runtimes flank the shared library: client on the left, core in the middle, server on the right. The arrangement is purely for readability; the dependency edges themselves are what matter.

9.2 The dashed non-edge

The dashed line between client and server is the most important element on the diagram. It records the explicit absence of a dependency. The Maven dependency graph forbids this edge, and a code review will reject any pull request that introduces it. Cross-runtime communication happens only over the wire.

9.3 Sub-packages inside core

Inside the middle column, bomberman-core is broken into seven sub-packages stacked from outer wire layer down to inner simulation. The net and net.dto packages sit at the top because they describe the public contract; the sim, entity, world, and geom packages sit below because they implement the deterministic model. GameConfig sits at the very root: every other package may depend on it, but it depends on nothing.

9.4 Why the rule matters

Without the layering rule, a small refactor in either runtime would risk pulling shared code in by accident, and the two runtimes would slowly drift into a state where they could only run together. With the rule, either runtime can be deleted and rewritten without touching the other.

10. Use case diagram

10.1 Five actors

The Player is the human user, two to four per match. The AI Bot may substitute for any player (the brief requires this fallback). The Game Master configures the arena and starts the match; in the default deployment this role is performed by an administrator. The Match Server appears as a system actor on the right because some use cases are owned by it. The Registry — the brief's Verwaltungsserver — brokers discovery.

10.2 Five lifecycle groups

The use cases are grouped by where in the player's lifecycle they fire: discovery (browse servers, log in, register), lobby (join a match, configure the arena, start the match), in-match (player actions on the top row, server actions on the bottom row), post-match (show ranking, start a new round, persist score), and cross-cutting (spectate, switch to AI, log gameplay).

10.3 Stereotypes that matter

Three dashed arrows carry stereotypes. Resolve chains «includes» Place bomb because every place triggers the chain-resolution pass on the server. Detonate inactive «extends» Place bomb because the brief's 30-second-no-response rule is modelled as a server-driven bomb placement on the laggard. Switch to AI «extends» Login because the AI fallback is offered immediately after the player logs in.

10.4 What the diagram tells the reader

The use case diagram fixes the system boundary. Anything not inside the dashed rectangle is external. Anything inside is something the system is responsible for. Reviewers can use this single view to check whether every functional requirement in the brief is covered by at least one use case.

Activity diagram · match lifecycle10 / 18

11. Activity diagram — match lifecycle

11.1 Three lanes

The activity is partitioned into three swimlanes. The Server lane owns boot, registration with the registry, and the tick loop's server side. The Shared lane in the centre carries the coordination steps that depend on both sides — quorum, arena initialisation, and the end-condition decision. The Client lane on the right owns boot, registry query, server selection, and the tick loop's client side.

11.2 The quorum gate

The match cannot start until the quorum gate (top centre diamond) is satisfied: at least two players, at most four. The gate is server-driven; the client's lobby view reflects the server's lobby state, never the other way around.

11.3 The fork into the tick loop

The gold bar is a fork. Server and client run their tick-loop activities in parallel from this point. Cross-lane dashed arrows are network packets: MOVE and BOMB_PLACE from client to server, SYNC and EXPLOSION from server to client.

11.4 The end condition

The bottom diamond evaluates the canonical end condition every tick: one or zero bombermen alive, or the match time has elapsed. If the answer is "no", control loops back into the tick. If "yes", control flows up to the ranking screen.

11.5 What the diagram does not show

The diagram intentionally omits reconciliation and prediction — those are activities inside the client tick-loop boxes (predict + render, reconcile on diff). They live in the server-authority page (page 13) so the lifecycle diagram stays a single-screen overview.

Sequence diagram · match start11 / 18

12. Sequence diagram — match start

12.1 Connect and identify

The sequence begins with a WebSocket upgrade. Immediately after, the client sends a JOIN envelope carrying the player's display name and age class. The server validates both, allocates a session identifier, and answers with the first LOBBY snapshot describing the current seats.

12.2 Lobby coordination

While the lobby is open, any change to its state — a player joining, leaving, marking themselves ready, the game master picking an arena — produces another LOBBY packet. The client's lobby view re-renders from that single source of truth.

12.3 Transition to active

When quorum is met and the game master starts the match, the server transitions MatchState from WAITING to STARTING, broadcasts a MATCH_STATE packet, initialises the arena, and transitions to ACTIVE. The first SYNC packet follows immediately and the client shows the arena scene.

12.4 What is not shown

The diagram does not show the bot-substitution path (any client can offer a non-random AI; the substitution is internal to the client). It also does not show heartbeats, which run in the background at 0.2 Hz from the moment of connection.

12.5 Why this is a reference

This sequence is the canonical first interaction for every practicum group's client. A new client implementation can verify its match-start behaviour against this single page; anything that diverges from it will not interoperate.

13. Sequence diagram — bomb placement

13.1 Press to wire

The player presses the place-bomb key. The client's input pump emits a BOMB_PLACE envelope with the current client tick number and the player's current cell. The envelope is queued on the WebSocket for delivery.

13.2 Server validates

On the next server tick, the envelope is drained. The server checks that the player still has at least one bomb in their pool and that the cell does not already hold a bomb or a wall. If either check fails, the request is dropped silently — the next SYNC tells the client what is real.

13.3 The fuse runs

A successful placement creates a Bomb entity with the configured fuse length. Every subsequent server tick, the fuse counts down. When the fuse reaches zero, the bomb transitions from TICKING to EXPLODING. Any neighbouring ticking bomb caught in the blast also transitions to EXPLODING on the same tick — the chain reaction.

13.4 Cosmetic vs canonical

Detonation produces two outbound packets. The EXPLOSION packet is cosmetic; it carries the list of tiles in the cross-shaped blast and the identifiers of any players eliminated. Clients use it to drive particles, camera shake, and the kill feed. The SYNC packet on the same tick carries the authoritative state — updated player health, the bomb removed, the destructible wall now gone.

13.5 What the client never trusts

The client never trusts its own prediction. If the predicted bomb is rejected by the server, the next SYNC restores the world as if the bomb were never placed and the buffered input is replayed without it.

Packet protocol13 / 18

14. Packet protocol

The wire format is JSON over WebSocket. Every message is wrapped in one envelope so framing, sequencing, and version negotiation stay in one place. We treat the protocol as a published contract because the brief requires cross-group interoperability.

14.1 Envelope

Every message is a JSON object with five fields: v (protocol version, uint8), t (message type tag), seq (monotonic sequence number from the sender, uint32), ts (sender epoch ms, uint64, for round-trip estimation only), and p (typed payload).

14.2 Catalogue

TypeDirectionTriggerRate
JOINClient → ServerFirst frame after connectOne per connection
MOVEClient → ServerEach input frame60 Hz max
BOMB_PLACEClient → ServerPlayer requests a bomb4 per second max
SYNCServer → ClientEvery tick60 Hz
EXPLOSIONServer → ClientOn detonationOn event
MATCH_STATEServer → ClientLifecycle transitionOn transition
LOBBYServer → ClientSeat changeOn change
HEARTBEATBothLiveness check0.2 Hz
DISCONNECTBothClean teardownOnce

14.3 Anti-desync rules

  • Strict monotonic seq per connection; duplicates and out-of-order frames are dropped.
  • The server overrides any illegal client prediction in the next snapshot; the client reconciles.
  • Each snapshot carries enough state to rebuild the world without the previous snapshot.
  • The simulation clock is monotonic on the server; wall-clock jumps do not affect the tick.
  • Envelope timestamps outside ±30 s of the server clock are rejected.

Server authority & synchronisation14 / 18

15. Server authority and synchronisation

The server runs the only canonical simulation. The client predicts locally for responsiveness and reconciles against every server snapshot. Between snapshots, the renderer interpolates so motion stays smooth.

15.1 The 60 Hz tick loop

The server runs a fixed step loop at 60 Hz (16.67 ms per tick). Each tick has five stages: drain queued inputs, resolve player intents into the world, advance entity lifetimes, evaluate the end condition, serialise and broadcast the snapshot. We measure p99 at 13.6 ms, leaving about 3 ms of headroom.

15.2 Client prediction

The client applies its input locally on the same frame it sends the matching MovePacket. The input is kept in a ring buffer of 30 frames, keyed by client tick. The renderer draws the predicted state immediately; under normal conditions the player feels no input latency.

15.3 Reconciliation

Each arriving SyncPacket carries a server tick. The client compares the snapshot to its predicted state at the same tick. If they agree, the input buffer is trimmed. If they disagree, the local state is replaced by the snapshot and every buffered input newer than that tick is replayed on top, frame by frame.

15.4 Interpolation

The renderer keeps its display cursor about 100 ms behind the latest snapshot and interpolates linearly between the two most recent ones. The buffer only affects rendering; it does not feed back into prediction or reconciliation.

15.5 Invariants

  1. Determinism. Same seed, same input log, same state stream.
  2. The server is the sole writer of the canonical world.
  3. The simulation clock is monotonic, not wall-clock.
  4. Replay cost is bounded by the input-buffer size.
  5. Each event identifier is unique within a match.

16. AI bot and state management

The brief requires that every client offer a non-random AI fallback. State management uses explicit enumerations with guarded transitions, rather than boolean flags.

16.1 The AI bot

The bot is a pure function of the current world snapshot and the bot's own seat. Every tick it builds a 13×13 threat map where each cell scores 0 (safe) to 4 (lethal this tick). The score is the maximum of three contributions: bomb proximity, current explosion membership, and predicted chain-reaction expansion.

A six-node decision tree sits on top: escape if the current cell is threat ≥ 3, chain-evade if a chain is about to spread into the player's row, pickup if one is reachable safely, destroy wall if a destructible wall is adjacent and clear, pressure enemy if an enemy is reachable along a low-threat path, otherwise wander.

Pathfinding is A* on the threat grid with edge cost 1 + threat[cell] and Manhattan distance as the heuristic; the closed set is bounded by the 169 cells of the grid.

16.2 State management

EnumerationStates
MatchStateWAITING → STARTING → ACTIVE → ENDING → ENDED
BombStateARMED → TICKING → EXPLODING → SPENT
PlayerStateJOINING → ALIVE → SPECTATING → LEFT

Each enumeration exposes a can(next) method so that the guard table lives next to the states. This pattern replaces multiple boolean fields with one enumeration and removes the possibility of impossible combinations such as "not alive and not spectating".

17. Prescriptive architecture

The system meets every functional requirement in the brief. The recommendations below are future-proofing rather than defect fixes. They are ordered by priority and accompanied by an effort estimate.

17.1 Must do

IDRecommendationEffort
P1Stand up the central registry (the brief calls it Verwaltungsserver). The client knows the server URL out of band today; cross-group interoperability requires a small REST service that match servers register against at boot and clients query at lobby time.About 2 days
P2Add Kryo serialisation as an optional protocol version two. JSON is fine for debugging; binary keeps bandwidth flat at higher player counts. The envelope shape does not change.About 3 days
P3Promote the inactivity-detonate rule to a first-class event. The brief requires that a player who fails to respond for 30 s is detonated by the server after a 3 s countdown; the logic currently lives in the connection layer.About 1 day

17.2 Should do

IDRecommendationEffort
R1Externalise GameConfig to a configuration file the server reads at boot. A game master can then pick arena size, bomb fuse, and pickup tables without a rebuild.Half a day
R2Add a pluggable arena registry — a folder of arena definitions the server loads at boot.About 1 day
R3Run the AI bot on the server as well as the client. The bot is already a pure function of the snapshot.About 1.5 days
R4Stream the gameplay log to an external sink (for example OpenTelemetry) in addition to the server window.About 1 day

17.3 Should not do

Three otherwise-tempting paths to avoid. First, adding a shared library between client and server beyond bomberman-core would dissolve the layering rule. Second, WebRTC peer-to-peer fragments the source of truth and breaks determinism. Third, moving the canonical simulation off the server thread sacrifices replayability for unneeded throughput.
