1. Overview and goals

BomberMen-X is a server-authoritative, distributed multiplayer arena game built as the capstone deliverable for the Software Architecture and Design (SAD) module of the M.Sc. Applied Computer Science programme at SRH University Stuttgart. The system is realised as a Maven three-module Java 17 reactor: bomberman-core (shared library), bomberman-server (Netty WebSocket server, port 8080), and bomberman-client (JavaFX desktop). The visual identity follows an Indian-festival mandala palette: teal (#008080), turmeric (#ffc107), henna (#ec407a), and gold (#daa520) on an aubergine (#150a1f) ground.

The architecture pursues five explicit qualities. Determinism: the server holds the only mutable GameWorld and advances it on a fixed 60 Hz tick, so the same input sequence produces the same outcome on any host. Fairness: clients render snapshots; they never simulate ahead of the server, removing both desync and a class of cheats. Clarity: every message on the wire is a typed Envelope with a MessageType discriminator and a Jackson-serialised payload, and the codec lives in exactly one place — com.bombermenx.core.net.WireCodec. Portability: build and run require only a JDK 17 and Maven 3.9, both available as zero-install portable drops. Demonstrability: the entire system stands up locally with one command via docker compose -f infra/docker-compose.yml up, and the JavaFX client connects without further configuration.

Within these qualities, the prototype intentionally constrains scope. There is no persistence layer; the 3D spectator view is deferred; OAuth is stubbed. Each deferral is documented in section 8 and was agreed with the supervisor at the Week 3 review.

3. Solution strategy

BomberMen-X adopts a three-tier client/server topology with a shared core library. The shared library, bomberman-core, is the only place where domain entities and wire DTOs are declared, so the server and the client cannot drift apart at compile time. The server is authoritative — it owns the only mutable instance of GameWorld, applies PlayerInput on tick boundaries, and emits a stream of WorldSnapshot objects derived by Snapshotter. The client renders snapshots and forwards local input frames; it never simulates predictively.

This strategy is deliberately conservative. A predictive client would reduce perceived input latency but introduces rollback, reconciliation, and divergence-handling concerns that have no place in an eight-week capstone. The 60 Hz authoritative tick is fast enough for the 13×11-tile arena and slow enough that the JSON wire format remains comfortable; profiling on the demo laptop shows the codec consuming under 3 percent of CPU during a match with four players and one bot.

The chosen wire format is JSON over WebSocket. Binary formats such as Protobuf were considered and rejected: the wire volume is dominated by snapshots whose redundancy collapses under WebSocket per-frame compression, and the readability of JSON has accelerated debugging during every weekly demo. WireCodec is the single Jackson entrypoint; every DTO that crosses the wire is exercised by WireCodecTest.

4. Building blocks

The reactor decomposes into three Maven modules and a fourth infrastructure tree.

4.1 bomberman-core (shared library)

Domain entities live under com.bombermenx.core.entity (Bomberman, Player, Score, Bomb, Explosion, PowerUpItem) and com.bombermenx.core.world (Arena, Tile, TileType, ArenaTheme, the abstract Bonus, and its seven concrete subclasses ArmorBonus, ExtraBombBonus, FlameBonus, KickBonus, LifeBonus, SpeedBonus, ThrowBonus). Geometry primitives — TilePos, Direction — live under com.bombermenx.core.geom. The simulation lives under com.bombermenx.core.sim (GameWorld, GameState, GameMode, Snapshotter). The wire protocol — Envelope, MessageType, WireCodec, and 24 DTOs — lives under com.bombermenx.core.net and com.bombermenx.core.net.dto.

4.2 bomberman-server (Netty WebSocket server)

BombServerApplication is the JVM entrypoint; WebSocketServer wires the Netty pipeline; GameServerHandler dispatches inbound frames by MessageType. MatchManager, Match, and MatchSession hold the authoritative simulation and drive the 60 Hz tick. LobbyService manages the pre-match flow and is fed by CosmeticsCatalog. Authentication routes through AuthRegistry, with DevAuthProvider for offline play and a stubbed GoogleAuthProvider ready for v0.3. ChatRouter and ProfanityFilter moderate text chat. BotController drives AI-filled slots through the same INPUT_FRAME path as a human. MetricsHandler exposes a Prometheus-style scrape on port 9091. SessionRegistry is the single source of truth for connected ClientSession objects.

4.3 bomberman-client (JavaFX desktop)

ClientLauncher bootstraps the JavaFX Application; SceneRouter swaps the active Scene between MainMenuView, LobbyView, ArenaView, and RankingsView. The painter is ArenaRenderer, working on a single JavaFX Canvas; the HUD is HudOverlay. MandalaArt and MandalaTheme apply the festival palette consistently across all four scenes. GameClient wraps the standard java.net.http.WebSocket and uses the same WireCodec as the server, eliminating any structural risk of DTO drift. AgeGate guards adult content. AudioBus, SpatialAudio, HapticsService, GamepadPoller, PostFx, CameraShake, and ParticleSystem are present as compile-clean stubs ready for v0.3.

4.4 infra/ (deployment tree)

infra/Dockerfile.server produces a slim runtime image of the server jar over eclipse-temurin:17-jre-jammy. infra/Dockerfile.client-build is a builder image that produces the client fat-jar in CI without polluting the developer machine. infra/docker-compose.yml stands up the server with ports 8080 (WebSocket) and 9091 (metrics). Helper scripts live under infra/scripts/.

5. Runtime

A typical match lifecycle proceeds as follows. A client connects to ws://localhost:8080. WebSocketServer accepts the handshake and binds a ClientSession in SessionRegistry. The client sends an AUTH_REQUEST; AuthRegistry consults the active AuthProvider and returns an AUTH_RESULT. The client then sends LOBBY_HELLO; LobbyService assigns a lobby slot, broadcasts LOBBY_STATE to peers, and replies with LOBBY_WELCOME. The player equips cosmetics through LOBBY_BUY and LOBBY_EQUIP; the lobby periodically pushes LOBBY_SNAPSHOT frames.

When the lobby reaches a quorum, MatchManager creates a Match and a MatchSession, broadcasts MATCH_START, and begins the 60 Hz tick. Each tick the session drains pending INPUT_FRAME frames, applies them to the GameWorld, advances bomb fuses and explosions, processes pickups, and asks Snapshotter for the current WorldSnapshot. The snapshot is serialised once by WireCodec and fan-out broadcast to every connected client in the match. Gameplay events (KILL_FEED, GAME_EVENT) are emitted as they happen. Bots are stepped by BotController immediately before input drain, so the rest of the tick treats them identically to humans.

The match ends when a victory condition fires (last Bomberman alive, score cap reached, or time-out). MATCH_END is broadcast; the MatchSession is reaped; players return to LobbyView. The whole lifecycle is observable through MetricsHandler, which exposes counters for connections, ticks, frames in, frames out, and match starts.

6. Deployment

Two deployment targets are supported.

Local desktop. The team's primary target for the prototype. Server is launched as a fat-jar (java -jar bomberman-server-0.1.0-SNAPSHOT.jar); client is launched via mvn javafx:run or its shaded fat-jar. This is the configuration that will be demonstrated on Tuesday 02 June 2026.

Cloud Run option (v0.3). infra/Dockerfile.server already produces a Cloud Run-compatible image (single port, stateless except for in-memory match state, sub-256 MB). The prototype does not require this path, but the deployment view in the slide deck calls it out so the examiners can see the architecture extends without rework.

8. Decisions and risks

The three load-bearing architecture decisions for the prototype are recorded as compact ADRs below.

  1. ADR-1: JSON over WebSocket as the wire format. Context: a single full-duplex channel per client is required; the protocol must be debuggable by hand during weekly demos. Decision: WebSocket over TCP, JSON payloads via Jackson, single WireCodec entrypoint. Consequences: text-frame readability accelerates debugging; per-frame compression keeps bandwidth acceptable; future binary upgrade is a single-module change because WireCodec is the only producer.
  2. ADR-2: Server-authoritative simulation, no client prediction. Context: cheating risk and a tight prototype budget. Decision: the server holds the only mutable GameWorld and ticks at 60 Hz; clients render WorldSnapshot objects exclusively. Consequences: input latency is one round-trip; rollback and reconciliation are not needed; BotController reuses the same INPUT_FRAME path, so the bot is structurally indistinguishable from a human at the server boundary.
  3. ADR-3: JavaFX desktop client, no browser target. Context: the team has Java expertise and no web-frontend bandwidth. Decision: JavaFX 17 over a single Canvas, SceneRouter for view changes. Consequences: native-quality rendering on the developer's host OS; binary distribution via a shaded fat-jar; no DOM/CSS rabbit holes; a future browser target would have to reimplement the client but could reuse bomberman-core verbatim if compiled to a JavaScript runtime.
