architecture

System Architecture

The three modules at a glance: bomberman-client (JavaFX) and the play.html browser client both speak WebSocket to bomberman-server on Cloud Run. The server holds match state and delegates persistence to Postgres (rankings) and Redis (sessions, rate-limit, pub/sub). The shared bomberman-core library sits beneath both runtime modules — the only piece of code that compiles into both ends of the wire.

sequence

Match Start

Trace of how a player goes from launching the desktop client to being inside a live 60 Hz match. WebSocket upgrade, HELLO/WELCOME, authentication via AuthRegistry, JOIN_LOBBY, the LOBBY_STATE countdown loop, and the eventual MATCH_START broadcast that begins GameWorld.tick at 60 Hz with SNAPSHOT messages flowing back to the client every frame.

sequence

Bomb Place → Detonate → Kill

End-to-end path of a single bomb placement: client INPUT(placeBomb=true) reaches MatchSession.applyInput, which calls GameWorld.placeBomb. Each tick the bomb counts down (fuse = 150 ticks). When the fuse hits zero detonate ray-traces the four axes, breaks soft walls, drops power-ups, and any player on a burning tile is killed. The resulting SNAPSHOT, KILL_FEED, EVENT and HAPTIC envelopes are pushed back to every client in the match. The explosion entity itself lives for 36 ticks.

deployment

Deployment View

Physical topology: clients on user machines, TLS termination at the Cloud Run edge, the server container sized for the free tier (512 MiB / 1 vCPU / max-instances 1), Cloud SQL Postgres 16 and Memorystore Redis 7 as managed sidecars (or self-hosted Redis on the same Compose stack for development), and the GitHub Actions CI cluster pushing images to Artifact Registry and triggering Cloud Run deploys via deploy-cloudrun.yml.
